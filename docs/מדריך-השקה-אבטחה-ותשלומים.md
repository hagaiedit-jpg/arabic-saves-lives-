# חלק 1 — חבילת שדרוג טכנית (קוד + הוראות)

המצב היום: אפליקציית-web בקובץ אחד (`index.html`), נתונים ב-Supabase, הקלטות ב-Cloudinary. המפתח הציבורי של Supabase והגדרות Cloudinary חשופים בקוד (זה תקין כשלעצמו — אבל מחייב הגנות בצד השרת). להלן ארבע ההגנות לפי סדר חשיבות.

## 1.1 🔴 Row-Level Security (RLS) ב-Supabase — הכי קריטי

זו ההגנה היחידה על הנתונים. כיום, כל מי שקורא את הקוד מחזיק במפתח ויכול לקרוא/לכתוב/למחוק. צריך מדיניות: הציבור קורא תוכן בלבד; כתיבה — רק למנהל מאומת.

**צעדים (בלוח הבקרה של Supabase → SQL Editor):**
```sql
-- 1. הפעלת RLS על טבלת התוכן
alter table phrases enable row level security;

-- 2. קריאה ציבורית של התוכן (אנונימי מותר לקרוא)
create policy "public read phrases"
  on phrases for select
  to anon, authenticated
  using ( true );

-- 3. כתיבה/עריכה/מחיקה — רק למשתמש מנהל מאומת
create policy "admin write phrases"
  on phrases for insert
  to authenticated
  with check ( auth.jwt() ->> 'email' = 'ADMIN_EMAIL@example.com' );

create policy "admin update phrases"
  on phrases for update
  to authenticated
  using ( auth.jwt() ->> 'email' = 'ADMIN_EMAIL@example.com' );

create policy "admin delete phrases"
  on phrases for delete
  to authenticated
  using ( auth.jwt() ->> 'email' = 'ADMIN_EMAIL@example.com' );
```
> החליפי `ADMIN_EMAIL@example.com` בכתובת המייל שאיתה תתחברי לניהול.
> אחרי הפעלה זו — פעולות הניהול באפליקציה יפסיקו לעבוד עד שתתחברי כמנהל (סעיף 1.2).

## 1.2 🔴 שער התחברות לניהול (Supabase Auth)

כל כפתורי ה"הוסף/ערוך/מחק/הקלט" צריכים לדרוש התחברות מנהל. Supabase Auth כבר זמין לך.

**צד שרת (לוח Supabase → Authentication):**
- צור משתמש מנהל אחד (Email + סיסמה חזקה).
- הפעל MFA (אימות דו-שלבי) למשתמש הזה.

**צד קוד (אני אכין לך כשתאשרי):** אוסיף לאזור הניהול:
```js
// התחברות מנהל — לפני שמציגים את כלי הניהול
async function adminLogin(email, password){
  const r = await fetch(SUPA_URL + '/auth/v1/token?grant_type=password', {
    method:'POST',
    headers:{'apikey':SUPA_KEY,'Content-Type':'application/json'},
    body: JSON.stringify({ email, password })
  });
  const d = await r.json();
  if(d.access_token){
    window._adminToken = d.access_token;       // נשמר בזיכרון, לא בקוד
    localStorage.setItem('admin_token', d.access_token);
    return true;
  }
  return false;
}
// וב-sbFetch של פעולות כתיבה — לצרף את הטוקן של המנהל במקום המפתח הציבורי:
//   'Authorization': 'Bearer ' + (window._adminToken || SUPA_KEY)
```
**התוצאה:** בלי התחברות מנהל — RLS חוסם כל כתיבה. גם אם מישהו מעתיק את הקוד, הוא לא יכול לשנות נתונים.

## 1.3 🟡 הגנת הקלטות — כתובות חתומות שפגות (signed + expiring URLs)

המטרה: שאי אפשר יהיה להוריד את ההקלטות או לשתף קישור — רק להאזין מתוך האפליקציה.

**הרעיון:** ההקלטות ב-Cloudinary יהיו "authenticated" (לא נגישות בכתובת ישירה). כשהאפליקציה רוצה להשמיע — היא מבקשת מ-Edge Function כתובת חתומה שתקפה ~5 דקות.

**Edge Function (Supabase → Edge Functions → `sign-audio`):**
```ts
// supabase/functions/sign-audio/index.ts
import { createHash } from "https://deno.land/std/hash/mod.ts";

Deno.serve(async (req) => {
  const { public_id } = await req.json();
  const cloud = Deno.env.get("CLOUDINARY_CLOUD")!;
  const apiKey = Deno.env.get("CLOUDINARY_API_KEY")!;
  const apiSecret = Deno.env.get("CLOUDINARY_API_SECRET")!; // סוד — רק כאן בשרת
  const expires = Math.floor(Date.now() / 1000) + 300;       // 5 דקות

  // חתימת auth_token של Cloudinary (HMAC-SHA256) — ראי תיעוד Cloudinary "Token-based authentication"
  const token = await cloudinarySignedToken(public_id, expires, apiSecret);
  const url = `https://res.cloudinary.com/${cloud}/video/authenticated/${public_id}?__cld_token__=${token}`;
  return new Response(JSON.stringify({ url }), { headers: { "Content-Type": "application/json" } });
});
```
**הגדרות שעלייך לבצע:**
1. ב-Cloudinary → Settings → Security → להפעיל **Token-based authentication** ולסמן את ה-preset כ-`authenticated`.
2. ב-Supabase → Edge Functions → Secrets: להגדיר `CLOUDINARY_CLOUD`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET`.
3. **מיגרציה של הקלטות קיימות:** ההקלטות הקיימות הן "public". צריך להמיר אותן ל-`authenticated` (דרך Cloudinary Admin API — אפשר באצ'). אני אכתוב סקריפט מיגרציה כשנגיע לזה.
4. צד קוד: במקום `new Audio(url)` → קודם לקרוא ל-`sign-audio`, לקבל כתובת זמנית, ואז לנגן.

> ⚠️ גם זה לא חוסם הקלטת-מסך תוך כדי האזנה (בלי DRM אין דרך) — אבל חוסם הורדה, שיתוף קישור, וגישה מחוץ לאפליקציה. זו הרמה המעשית הגבוהה ביותר.

## 1.4 🟡 נעילת ה-Upload Preset ב-Cloudinary
ה-preset `arabic_audio` הוא "unsigned" → כל אחד יכול להעלות לחשבון שלך.
- לעבור ל**signed uploads** (חתומות ע"י ה-Edge Function), **או** להגביל את ה-preset: פורמטים מותרים בלבד, גודל מקסימלי, תיקייה קבועה, מצב moderation.

## 1.5 🟡 Cloudflare מול האתר (חינם)
הוספת האתר ל-Cloudflare (החלפת ה-DNS) נותנת בחינם:
- **WAF** (חומת אש לאפליקציה), **הגנת DDoS**, **מיתון בוטים**, ו**הגבלת קצב** (rate limiting) — מענה ישיר ל-brute-force.
- צעדים: לפתוח חשבון Cloudflare → להוסיף את הדומיין → להעביר את ה-nameservers → להפעיל WAF + Rate Limiting rules על נתיבי הכתיבה.

## 1.6 🟢 גיבויים וניטור
- **Supabase → Database → Backups:** להפעיל גיבויים אוטומטיים (שחזור ממחיקה זדונית).
- **Sentry / Supabase Logs:** ניטור שגיאות והתראות.
- **Snyk / Dependabot:** סריקת חולשות (פחות רלוונטי לקובץ יחיד, יותר אם יוסיפו תלויות).

# חלק 2 — בטיחות ופרטיות: שלך ושל המנויים (חיילים)

זה הסעיף הכי קריטי עבורך, כי **חשיפת נתוני חיילים היא סיכון ברמה אחרת** — לא רק עסקי אלא ביטחוני.

## 2.1 ⭐ כלל הזהב: מינימום נתונים (Data Minimization)
**אל תאספי מידע שאינך חייבת.** ככל שתאגרי פחות — כך פחות יש מה לדלוף. עבור חיילים ספציפית:
- **אל** תאספי: שם מלא, מספר אישי, יחידה, תפקיד, בסיס, מיקום, תמונות.
- **כן** מינימום הכרחי למנוי: כתובת מייל (להתחברות), וסטטוס תשלום.
- שקלי **חשבונות אנונימיים/בדויים** — התחברות עם כינוי או מייל בלבד, בלי לקשור לזהות אמיתית.

## 2.2 ההקלטות של החיילים — להשאיר על המכשיר (כמו היום!)
כרגע ההקלטות שהחייל מקליט את עצמו נשמרות **רק במכשיר שלו** (IndexedDB) ולא עולות לשרת. **זה מעולה — שמרי על זה.** אל תעלי הקלטות אישיות של משתמשים לשרת; זה הופך אותך ל"מאגר קולי של חיילים" — סיכון מיותר.

## 2.3 מסמכים משפטיים שחובה שיהיו לך
1. **מדיניות פרטיות (Privacy Policy)** — מה נאסף, למה, איפה נשמר, למי מועבר, זכויות המשתמש, יצירת קשר.
2. **תנאי שימוש (Terms of Service)** — כולל הגבלת אחריות, זכויות יוצרים על התוכן, ביטולים/החזרים.
3. **הסכמה (Consent)** — checkbox בהרשמה לאישור המדיניות.
> את אלה כדאי שעו"ד ינסח, במיוחד בגלל קהל היעד.

## 2.4 חוק הגנת הפרטיות (ישראל) ורישום מאגר מידע
- אם תנהלי מאגר עם פרטי מנויים — ייתכן שתידרשי לעמוד בדרישות **חוק הגנת הפרטיות** ותקנות אבטחת המידע, ואולי **רישום מאגר מידע** אצל הרשם.
- חובות מרכזיות: אבטחת מידע סבירה, צמצום, מחיקה לפי דרישה, יידוע על דליפה.
> לאמת מול עו"ד פרטיות — הסף לרישום והדרישות תלויים בהיקף ובסוג המידע.

## 2.5 הצפנה
- **בתעבורה:** HTTPS — כבר יש לך. ודאי שכל הקריאות ב-HTTPS.
- **במנוחה (at rest):** Supabase ו-Cloudinary מצפינים נתונים בשרת. ודאי שאת לא שומרת מידע רגיש ב-localStorage של המשתמש מעבר להכרחי.

## 2.6 ⚠️ הערת OPSEC — מכירה לחיילים
מכירה של אפליקציה שמרכזת משתמשים מצה"ל עשויה להיות כרוכה בשיקולים ביטחוניים/רגולטוריים:
- שקלי אם נדרש **אישור/תיאום** כלשהו (למשל אם משווקים דרך גורם צבאי, או אם נאסף מידע שעלול להצטבר לתמונת מודיעין).
- אל תאפשרי הצטברות מידע שמאפשר זיהוי יחידות/פעילות (למשל: סטטיסטיקות שימוש לפי מיקום).
> זו נקודה להעלות מול עו"ד **וגם** מול גורם שמכיר רגולציה ביטחונית.

## 2.7 תוכנית תגובה לאירוע (Breach Response)
מסמך קצר שמכין מראש: מי מודיע, למי (רשם/משתמשים), תוך כמה זמן, איך עוצרים את הדליפה, איך מאפסים מפתחות. עדיף להכין בשקט מראש מאשר בלחץ.

## 2.8 הסכמי ספקים (DPA)
מול Supabase, Cloudinary וספק הסליקה — לוודא שיש **Data Processing Agreement** ושהם עומדים בתקני אבטחה (SOC 2 / ISO 27001). כל השלושה הגדולים עומדים בזה.

# חלק 3 — גביית תשלום (סליקה) ומודל מנוי

**כן, צריך מערכת סליקה.** את לעולם **לא** שומרת פרטי כרטיס בעצמך (זה דורש תקן PCI-DSS כבד) — ספק הסליקה מארח את הצ'ק-אאוט ומחזיק את הנתונים.

## 3.1 שלוש דרכים אפשריות
| גישה | מתאים ל | יתרונות | חסרונות |
|---|---|---|---|
| **Merchant of Record** (Paddle, Lemon Squeezy) | מכירה גלובלית, יזם/ית יחיד/ה | הם **מטפלים במע"מ/מס/חשבוניות** בכל העולם, מנויים מובנים, פשוט | עמלה מעט גבוהה יותר |
| **סולק ישראלי** (Cardcom, PayPlus, Tranzila, Grow/Meshulam, Hyp/PayMe) | מכירה בעיקר בישראל | עמלות נמוכות, שקלים, תמיכה בעברית, חשבונית ישראלית | את אחראית למע"מ/חשבונית, אינטגרציה מורכבת יותר |
| **חנויות אפליקציות** (Apple/Google) | אם תהפכי לאפליקציה נייטיב עטופה | תשתית תשלום מלאה, אמון משתמשים | עמלה 15–30%, צריך אריזה כ-app |

## 3.2 ההמלצה למצב שלך
- **PWA + מכירה בעיקר בישראל:** סולק ישראלי כמו **PayPlus / Cardcom / Grow** — תשלום בשקלים, חשבונית, מנויים חוזרים. דורש **עוסק מורשה/חברה**.
- **אם תרצי פשטות מקסימלית וזירה גלובלית:** **Paddle** או **Lemon Squeezy** (Merchant of Record) — הם הופכים להיות "המוכר" הרשמי ומטפלים במס/מע"מ/חשבוניות במקומך. אידיאלי ליזמת יחידה שלא רוצה להתעסק עם רגולציית מס.

## 3.3 מה צריך מולם
1. **ישות עסקית:** עוסק פטור/מורשה או חברה (רואה חשבון יכוון מה מתאים לפי היקף צפוי).
2. **חשבונית ומע"מ:** סולק ישראלי → את מנפיקה חשבונית (אפשר עם iCount/חשבונית ירוקה). MoR → הם מנפיקים.
3. **דף תשלום מאורח (hosted checkout):** לא לאחסן כרטיסים אצלך.
4. **מנוי חוזר (recurring):** הספק מנהל חיוב חודשי/שנתי, ניסיון חינם, וביטול.

## 3.4 איך מחברים תשלום לגישה בתוכן (טכני)
זהו ה"חיבור" שמפריד בין מנוי משלם לחינמי:
```
משתמש משלם → הסולק שולח Webhook ל-Edge Function שלך →
ה-Function מסמן ב-Supabase: user_id => subscription_active = true (עד תאריך) →
האפליקציה בודקת את הסטטוס ופותחת/חוסמת תוכן בהתאם (דרך RLS).
```
- **טבלת `subscriptions`** ב-Supabase (user_id, status, valid_until).
- **RLS על תוכן פרימיום**: `using ( exists(select 1 from subscriptions where user_id = auth.uid() and status='active' and valid_until > now()) )`.
- כך גם אם מישהו מעתיק את הקוד — בלי מנוי פעיל ה-RLS לא נותן את התוכן.
